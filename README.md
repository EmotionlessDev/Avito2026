## Генерация слотов

### Выбранный подход: Lazy generation
Слоты генерируются **в момент запроса** пользователем доступных слотов на конкретную дату, а не заранее.

### Как это работает:
1. Пользователь запрашивает слоты на 2026-04-01
2. Сервис проверяет: слоты уже существуют в БД?
3. Если да, то просто возвращает существующие слоты
4. Если нет, то генерирует слоты на эту дату, после чего возвращает их

### Почему этот подход лучше для данного проекта
1. **Экономия места в БД**
   - Не храним слоты на даты, которые никто не запросит
   - С ленивой генерацией: только реально запрошенные даты

2. **Пользователь всегда делает GET перед бронью**
   - Невозможно забронировать слот без предварительного получения списка
   - Значит первый запрос всё равно произойдёт → генерация оправдана

## Структура и консистентность данных

В проекте используется **PostgreSQL**:

- **Уникальные поля и индексы** гарантируют консистентность:
  - `users.email` уникален, чтобы исключить дубликаты пользователей.
  - `schedules.room_id` уникален, чтобы для одной комнаты было только одно расписание.
  - `slots` имеет unique `(room_id, start_time, end_time)`, предотвращая пересечение слотов.
  - `bookings` имеет **частичный уникальный индекс** `unique_active_booking_per_slot`, позволяющий одной активной бронью занимать слот.
- **Индексы и внешние ключи** ускоряют выборки и поддерживают целостность:
  - все связи `room_id`, `schedule_id`, `slot_id`, `user_id` защищены через `FOREIGN KEY` с `ON DELETE CASCADE`.
  - индексы позволяют быстро фильтровать по активным броням и искать слоты/расписания.

## Интеграционное тестирование с Testcontainers

В проекте для интеграционных тестов используется библиотека **[Testcontainers](https://github.com/testcontainers/testcontainers-go)**.  

Основная цель — поднимать **временную PostgreSQL базу данных** для тестов, чтобы:

- Тесты не зависели от локальной базы разработчика.
- Можно было безопасно проверять работу с реальной БД, включая связи, индексы и ограничения.
- Каждое тестирование начиналось с "чистой" базы и гарантированно завершалось очисткой ресурсов.

Таким образом, интеграционные тесты полностью изолированы и воспроизводимы, что повышает надежность и стабильность тестирования бизнес-логики.
